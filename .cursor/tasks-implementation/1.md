Plan implementacji tasku people

Cel

Zbudować endpoint Fastify, który uruchamia pipeline dla zadania people: pobiera CSV z Hubu, filtruje rekordy po kryteriach biznesowych, batchowo taguje job przy pomocy LLM ze Structured Output, wybiera osoby z tagiem transport, wysyła wynik na https://hub.ag3nts.org/verify i zwraca odpowiedź Hubu.

Obecny stan projektu





src/app.ts rejestruje dziś tylko plugin Supabase, więc task trzeba dopiąć ręcznie przez app.register(...).



src/server.ts to prosty bootstrap serwera; nie ma jeszcze warstwy route/task.



src/plugins/supabase.ts pokazuje obowiązujący wzorzec pluginów Fastify.



src/types/fastify.d.ts jest właściwym miejscem na rozszerzenie FastifyInstance o nowe dekoratory.



src/tasks/1/index.ts i pozostałe katalogi tasków są placeholderami, więc people trzeba zaprojektować od zera w src/tasks/<nr>/.



package.json nie zawiera jeszcze SDK OpenAI ani parsera CSV.

Wymagania jawne i niejawne





Pobranie people.csv z URL z kluczem API.



Filtrowanie do osób spełniających wszystkie warunki:





gender === "M"



wiek w 2026 roku między 20 a 40 lat włącznie, czyli born w zakresie 1986..2006



city odpowiada Grudziądz po normalizacji



Klasyfikacja pola job tylko do dozwolonych tagów: IT, transport, edukacja, medycyna, praca z ludźmi, praca z pojazdami, praca fizyczna.



Jedna osoba może mieć wiele tagów, ale finalnie wybieramy tylko rekordy zawierające tag transport.



born w payloadzie końcowym musi być liczbą całkowitą, a tags tablicą stringów.



Rozwiązanie powinno rozdzielać HTTP, orkiestrację tasku, parsing CSV i integrację z LLM.



Należy użyć Structured Output, a nie ręcznego parsowania odpowiedzi modelu.

Proponowany przepływ

flowchart LR
routeHandler --> orchestrator
orchestrator --> downloadCsv
orchestrator --> parseCsv
parseCsv --> normalizeRows
normalizeRows --> filterBaseCriteria
filterBaseCriteria --> dedupeJobs
 dedupeJobs --> batchTagging
batchTagging --> mergeTags
mergeTags --> selectTransport
selectTransport --> buildVerifyPayload
buildVerifyPayload --> verifyHub
verifyHub --> routeResponse

Podział odpowiedzialności





Warstwa HTTP:





route Fastify tylko waliduje wejście, odpala serwis i zwraca wynik



Logika biznesowa:





filtr wieku/płci/miasta, selekcja tagu transport, budowa odpowiedzi



Integracje techniczne:





pobranie CSV przez fetch



wywołanie OpenAI przez plugin



wysyłka do Hub /verify



Schematy:





osobno dla Fastify i osobno dla Structured Output, żeby nie mieszać kontraktu HTTP z kontraktem LLM

Dane wejściowe, wyjściowe i transformacje





Wejście z Hub CSV:





surowe rekordy z polami co najmniej name, surname, gender, born, city, job



Transformacje:





parsing CSV



trim i normalizacja tekstu



bezpieczna konwersja born do number



filtr po gender, born, city



deduplikacja identycznych job przed wysyłką do LLM



batch tagging unikalnych zawodów



rekonstrukcja tags na rekordach osób



filtr końcowy po transport



Wyjście do Hub:





{ apikey, task: 'people', answer: PersonAnswer[] }



Wyjście endpointu Fastify:





przy dryRun: true: wyliczeni kandydaci bez wysyłki do Hub



normalnie: kandydaci + odpowiedź verify

Integracja z OpenAI





Rekomendowany model: gpt-4.1-mini





wystarczy do klasyfikacji krótkich opisów stanowisk



dobrze nadaje się do Structured Output i batchowania przy niższym koszcie



Fallback dla niejednoznacznych wyników lub słabej jakości klasyfikacji: gpt-4.1



Prompt powinien zawierać:





definicje wszystkich tagów



zasadę, że wolno używać wyłącznie tagów z listy



zasadę, że można zwrócić wiele tagów dla jednego zawodu



numerowane/identyfikowalne rekordy wejściowe



Structured Output schema powinno wymuszać format w rodzaju:





[{ jobId: string, tags: PeopleTag[] }]



Optymalizacja:





taguj tylko rekordy po wstępnym filtrze biznesowym



taguj unikalne job, a wynik mapuj z powrotem na osoby



dziel na batch-e, np. po 20-50 opisów, jeśli CSV okaże się większy

Walidacja i obsługa błędów





W src/config/env.ts fail-fast przy brakujących kluczach.



Przy błędnym CSV:





jeśli cały plik jest nieparsowalny, zwróć błąd zadania



jeśli pojedynczy rekord ma brakujące pola, pomiń go i zaloguj licznik odrzuceń



Przy LLM:





jeśli Structured Output nie przejdzie walidacji, traktuj to jako błąd integracji, nie zgaduj tagów ręcznie



Przy /verify:





sprawdzaj status HTTP i treść odpowiedzi



zachowaj surową odpowiedź w payloadzie endpointu do debugowania



Dla miasta użyj porównania odpornego na wielkość liter i brak polskich znaków, żeby objąć np. Grudziadz vs Grudziądz.



Rozważ deduplikację osób po kluczu name + surname + born + city + job, jeśli CSV może zawierać duble.

Minimalny kontrakt endpointu





POST /tasks/people/run



request body:





opcjonalne { dryRun?: boolean }



response body:





ok



sourceCount



eligibleCount



transportCount



answer



verifyResponse tylko gdy nie jest dryRun

Testowanie





Ponieważ repo nie ma jeszcze test runnera, sensowny minimalny poziom to:





test manualny endpointu z dryRun



logi/liczniki na etapach: pobrane rekordy, po filtrze bazowym, liczba unikalnych job, po tagowaniu, finalna liczba transport



Jeśli chcesz dodać testy automatyczne, najlepszy pierwszy zakres to czyste funkcje:





normalizacja miasta



wyliczanie wieku z born



filtr kandydatów



mapowanie batchowych wyników LLM na osoby

Założenia





Numer katalogu dla tasku people nie został podany, więc plan zakłada implementację w src/tasks//index.ts i powiązanych plikach.



Zakładam, że klucz AG3NTS do pobrania CSV i apikey do /verify są tym samym sekretem lub pochodzą z tego samego źródła konfiguracyjnego; jeśli w praktyce są różne, rozdziel je w env na dwa pola.



Zakładam implementację jako endpoint Fastify zgodnie z Twoją decyzją, ale cała logika powinna pozostać niezależna od HTTP, żeby dało się ją później odpalić także z CLI lub crona bez przepisywania kodu.
